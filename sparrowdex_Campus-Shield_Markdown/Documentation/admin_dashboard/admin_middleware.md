The Admin Middleware component in the Campus-Shield project is a crucial part of the authentication and authorization system, specifically designed to protect routes that require administrative privileges. This middleware ensures that only users with the 'admin' role can access certain API endpoints and perform administrative actions.

### Core Functionality

The admin middleware is implemented in the `server/middleware/admin.js` file. Its primary function is to check if the authenticated user has the 'admin' role before allowing access to protected routes. Here's the implementation:

```javascript
const admin = (req, res, next) => {
  if (req.user.role !== 'admin') {
    return res.status(403).json({
      success: false,
      message: 'Access denied. Admin privileges required.'
    });
  }
  next();
};

module.exports = admin;
```

<artifact ArtifactUUID="2b6ad7f0-57fa-40a9-978b-f02c6bd48353">Role-Based Access Control</artifact>

The middleware performs a simple but effective role check:
- It assumes that the `req.user` object has been populated by a previous authentication middleware.
- It checks if the user's role is 'admin'.
- If the role is not 'admin', it returns a 403 Forbidden response with a clear error message.
- If the role is 'admin', it calls `next()` to allow the request to proceed to the next middleware or route handler.

### Integration with Routes

The admin middleware is typically used in combination with the authentication middleware to protect admin-only routes. In the `server/routes/admin.js` file, we can see how it's applied:

```javascript
const express = require('express');
const auth = require('../middleware/auth');
const admin = require('../middleware/admin');

const router = express.Router();

router.get('/stats', auth, admin, async (req, res) => {
  // Admin-only route logic
});

module.exports = router;
```

<artifact>Middleware Chaining</artifact>

This pattern of chaining middlewares (`auth` followed by `admin`) ensures that:
1. The user is authenticated (handled by the `auth` middleware).
2. The authenticated user has admin privileges (handled by the `admin` middleware).

### Error Handling and Security

The admin middleware contributes to the overall security of the application by:
- Providing a clear and consistent access denial message.
- Using HTTP status code 403 (Forbidden) to indicate that the server understood the request but refuses to authorize it.
- Keeping the middleware function simple and focused, adhering to the Single Responsibility Principle.

### Extensibility

While the current implementation is straightforward, it can be easily extended to support more complex role-based access control:
- Additional roles could be added (e.g., 'moderator', 'super-admin').
- The middleware could be modified to accept an array of allowed roles.
- It could be integrated with a more sophisticated permissions system for granular access control.

<artifact ArtifactUUID="6f89c003-764a-4ef3-a44a-c66d9637ebd1">Admin Request Workflow</artifact>

The admin middleware plays a crucial role in the admin request workflow, which is implemented across several files:
- `client/src/pages/RequestAdmin.tsx`: Allows users to request admin access.
- `server/models/AdminRequest.js`: Defines the schema for admin requests.
- `client/src/pages/AdminRequests.tsx`: Provides an interface for reviewing admin requests.
- `server/routes/admin.js`: Implements the API endpoints for managing admin requests.

This workflow ensures that the admin role is granted through a controlled process, maintaining the security and integrity of the administrative functions within the Campus-Shield application.
